常见问题

关于 Clipal 的常见问题解答

Clipal 是本地工具还是云端代理？

两者都不是。Clipal 是一个独立的本地二进制文件。你的 API Key 存储在本地被加密的 config.json 中，绝不经过任何云端，保证 100% 数据安全。

我能在 Claude Code 和基于 OpenAI 的工具中配合 OpenRouter 或 new-api 使用 Clipal 吗？

当然可以。Clipal 原生支持 OpenAI 兼容协议、Anthropic Claude 以及 Google Gemini 协议。你可以将终端里的 Claude Code、基于 OpenAI 的 Codex 以及 Gemini CLI 流量，轻松转发至任意第三方提供商或本地端点。

Clipal 如何处理三方中转 API 的 429 (Too Many Requests) 限流报错？

Clipal 提供多维度的可用性保障。除了单 Provider 的多密钥轮询外，我们还支持多 Provider 并发调度与智能会话粘性。如果你的主力 OpenRouter 密钥触发限流，网关会毫秒级切换至其他密钥或备用提供商，并强制执行会话粘性逻辑，确保你的编码工作流永不中断。

在 Codex 中使用 Clipal 会影响我的聊天历史记录或会话稳定性吗？

不会，反而会更稳。针对 Codex / OpenCode，Clipal 采用标准化的 model_provider = "clipal" 配置。不同于其他频繁更换 Provider 标识（这会导致会话历史重置）的方案，Clipal 始终保持 Provider 标识一致，仅在后台无感切换 API 密钥或端点，从而完美保留你的所有上下文和历史记录。

Clipal 与 cc-switch 有什么区别？

cc-switch 主要负责多配置切换，而 Clipal 是高性能本地网关，不仅支持 settings.json 的深度注入与接管，还支持复杂的多密钥 Token 池管理和实时请求观测。

Clipal 能解决 Bedrock 的 "Invalid API Key" 报错吗？

可以。绝大多数代理在处理 Claude Code 连接 Bedrock/Vertex 时会因为 Header 校验失败。Clipal 提供原生协议仿真，从 Socket 层级修复此问题。

运行性能如何？

极度轻量。使用 Go 开发，内存占用低于 50MB，内部协议转发延迟低于 1 毫秒，远比基于 Node.js 或 Python 的代理更高效。

Clipal 如何保护我已有的配置文件？

不同于其他工具直接覆盖整个 settings.json，Clipal 采用非破坏性合并策略。我们仅更新必要的 Provider 字段，并完整保留你的 enabledPlugins 和自定义环境变量。

我厌倦了 Claude Code 的登录循环，Clipal 能解决吗？

是的。登录循环通常是由代理时 Header 处理不当引起的。Clipal 的协议仿真层确保护理所有 Header 注入，让你的会话保持稳定，无需频繁重新认证。

我如何知道代理是否真的在工作？

Clipal 提供实时观测面板。点击即可查看完整的请求流、响应时间和错误代码。如果请求失败，我们会告知是网络超时还是 API Key 无效。
