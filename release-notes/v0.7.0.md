## v0.7.0

### English

#### Highlights

- Added `/clipal` as the preferred unified local ingress for Claude-, OpenAI-, and Gemini-style traffic.
- Reworked request classification around explicit protocol families and capabilities instead of loose path heuristics.
- Split failover cursors by capability so special routes like OpenAI `responses` and Gemini streaming no longer perturb the default traffic path.
- Added session-sticky busy overflow routing so concurrency-limit `429` responses can wait briefly, retry once, and then spill to another provider without poisoning provider health.
- Isolated advisory `count_tokens` flows so they no longer mutate provider health state or shift primary routing.
- Improved runtime and Web observability with scope-aware current provider snapshots, capability-aware request status, and expanded regression coverage.

#### What Changed

- Added: unified `/clipal/*` routing with explicit protocol-to-client dispatch
- Added: capability-based request context propagation for routing, auth handling, and status reporting
- Added: scoped routing state for default traffic, OpenAI `responses`, and Gemini `streamGenerateContent`
- Added: sticky affinity learning for explicit response chains and short-lived multi-turn human-message features
- Added: provider busy backpressure with bounded inline wait, retry probe, and overflow rebinding
- Improved: provider auth injection now preserves the client's explicit auth carrier when forwarding through gateways
- Improved: unknown unified-ingress paths now fail fast with explicit `/clipal` endpoint errors
- Improved: Web status output now exposes scope-specific current providers and user-visible capability metadata
- Improved: routing and protocol tests now cover unified ingress, scoped failover, auth-carrier preservation, and compatibility aliases
- Changed: Claude and Gemini `count_tokens` requests are treated as advisory single-shot passthroughs

### 中文

#### 亮点

- 新增 `/clipal` 作为 Claude、OpenAI、Gemini 风格请求的统一本地入口。
- 请求分类逻辑改为基于显式协议族和能力识别，而不是依赖宽松的路径启发式判断。
- 故障切换游标按能力拆分，像 OpenAI `responses` 和 Gemini 流式请求这类特殊链路不会再扰动默认主流量路径。
- 新增 session-sticky busy overflow routing：遇到并发限制类 `429` 时，先短暂等待并重试一次，再受控溢出到其他 provider，而不是直接把 provider 判坏。
- `count_tokens` 这类 advisory 请求被隔离，不再修改 provider 健康状态，也不会推动主路由游标漂移。
- 运行时和 Web 可观测性增强，支持按 scope 查看当前 provider、请求 capability，并补齐了大批回归测试。

#### 变更内容

- 新增：统一 `/clipal/*` 路由入口，并基于协议显式分发到对应 client 池
- 新增：基于 capability 的请求上下文传递，用于路由、鉴权和状态展示
- 新增：为默认流量、OpenAI `responses`、Gemini `streamGenerateContent` 分别维护独立路由状态
- 新增：基于显式 response 链路和短期多轮人类消息特征的黏性学习
- 新增：provider busy 背压处理，支持 bounded inline wait、恢复探测和 overflow 重绑
- 改进：转发到网关类上游时，provider 鉴权注入会尽量保留客户端原本使用的认证载体
- 改进：统一入口上的未知路径会更早失败，并返回明确的 `/clipal` 端点错误
- 改进：Web 状态页现在可展示按 scope 的当前 provider 与面向用户的 capability 信息
- 改进：路由与协议测试覆盖统一入口、按 scope 故障切换、认证载体保留以及兼容别名
- 调整：Claude 和 Gemini 的 `count_tokens` 请求改为 advisory 的 single-shot passthrough
