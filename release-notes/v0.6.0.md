## v0.6.0

### English

#### Highlights

- Providers now support multiple API keys via `api_keys`, in addition to the original single `api_key` form.
- In auto mode, Clipal now retries the next available key within the same provider before failing over to the next provider.
- Manual mode remains strict pass-through for the pinned provider and no longer changes key/provider selection behind the scenes.
- Provider runtime status and Web UI now show configured key count, available key count, and an explicit "no keys available" state.
- Half-open circuit-breaker probes are now released correctly when a provider exhausts all currently available keys.
- The Web UI provider form now accepts one key per line, saving a single line as `api_key` and multiple lines as `api_keys`.

#### What Changed

- New: support `api_keys` in provider config while keeping backward compatibility with `api_key`
- Improved: per-provider key retry before cross-provider failover
- Fixed: restored strict manual-mode semantics for pinned providers
- Improved: clearer runtime/UI visibility into key availability
- Fixed: half-open circuit breaker probe leak when all keys for a provider are temporarily unavailable
- Improved: Web UI create/edit flow for multi-key providers

#### Docs

- Updated README examples and field descriptions for `api_key` / `api_keys`
- Updated Web UI documentation for multi-key entry and key-availability status
- Updated platform setup docs to mention `api_key` or `api_keys`

#### Notes

- Each provider must set either `api_key` or `api_keys`
- `api_keys` is an ordered list
- In manual mode, Clipal still returns the pinned provider response directly

### 中文

#### 亮点

- Provider 现已支持通过 `api_keys` 配置多个 API Key，同时保留原有单个 `api_key` 的配置形式。
- 在自动模式下，Clipal 现在会先在同一个 provider 内重试下一个可用 key，再切换到下一个 provider。
- 手动模式继续保持对 pinned provider 的严格透传，不会在背后自动改变 key 或 provider 选择。
- Provider 运行态与 Web UI 现在会显示已配置 key 数、当前可用 key 数，以及明确的“无可用 key”状态。
- 当某个 provider 的全部 key 当前不可用时，半开状态的熔断器探测现在会被正确释放。
- Web UI 的 provider 表单现已支持“一行一个 key”；单行保存为 `api_key`，多行保存为 `api_keys`。

#### 变更内容

- 新增：provider 配置支持 `api_keys`，同时保持对 `api_key` 的向后兼容
- 改进：在跨 provider failover 之前，先进行 provider 内部的 key 重试
- 修复：恢复 pinned provider 的严格手动模式语义
- 改进：增强运行态 / UI 对 key 可用性的可见性
- 修复：当某个 provider 的全部 key 临时不可用时，修复半开状态熔断器探测泄漏问题
- 改进：优化 Web UI 中多 key provider 的创建 / 编辑流程

#### 文档

- 更新 README 中关于 `api_key` / `api_keys` 的示例与字段说明
- 更新 Web UI 文档，补充多 key 录入与 key 可用性状态说明
- 更新各平台安装文档，统一说明可使用 `api_key` 或 `api_keys`

#### 说明

- 每个 provider 必须配置 `api_key` 或 `api_keys` 二选一
- `api_keys` 是一个有序列表
- 在手动模式下，Clipal 仍会直接返回 pinned provider 的响应
